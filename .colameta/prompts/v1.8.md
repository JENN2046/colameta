Problem to fix:
The v1.7 materialized audit package reported validation_status_summary=passed and included `PASSED: python3 -m unittest discover -s tests`, while the executor report excerpt explicitly stated that `python3 -m unittest discover -s tests` failed with 38 socket PermissionError errors. This means report/audit normalization allowed a failed or inconsistent validation command to be summarized as passed.

Goal:
Harden executor validation reporting so failed command evidence cannot be normalized into PASS.

Source-of-truth rule:
1. Structured command records and exit codes are the primary source of truth.
2. If a command record has a non-zero exit_code, validation must not summarize as passed, even if its status string says PASSED.
3. If status and exit_code disagree, preserve an explicit inconsistency signal.
4. If the final executor report text contradicts structured validation summary for the same command, preserve validation_inconsistent rather than silently accepting the summary.
5. Audit packages and get_audit_package validation reads must expose the failed/inconsistent command evidence clearly enough for Commander review.

Required implementation behavior:
- Add or refactor pure helper logic, preferably in a small focused module if it keeps runner/executor_run_reports.py readable.
- Preserve backward compatibility for existing summary.validation_results string lists.
- Add structured validation command records to reports/audit packages, including at least:
  - command index or stable id
  - original command text
  - executed command text when available
  - status as recorded
  - derived status from exit_code
  - exit_code
  - stdout/stderr excerpts, redacted and bounded
  - inconsistency reason when status and exit_code disagree
- Derive validation_status_summary from structured evidence when available.
- When any command has exit_code != 0, validation_status_summary must be failed or inconsistent, never passed.
- When status says PASSED but exit_code is non-zero, validation_inconsistent must be true.
- When summary text says PASSED for a command but executor report text says the same command failed, validation_inconsistent must be true.
- Materialized executor-run audit package and version audit package must carry the new truth-source fields.
- get_audit_package(section="validation") should expose the structured command records or a bounded sample plus validation_inconsistent status, not only plain string samples.
- Markdown report output should include exit_code for validation commands when available.

Tests first:
Add focused tests before or alongside implementation. Cover at minimum:
1. A synthetic command record with status=PASSED and exit_code=1 produces validation_inconsistent=true and validation_status_summary not equal to passed.
2. A synthetic command record with status=FAILED and exit_code=1 is summarized as failed and preserves command, exit_code, stdout/stderr excerpts.
3. Existing string-only validation_results fallback still works for old reports.
4. A report with summary `PASSED: python3 -m unittest discover -s tests` but report text containing the same command as failed is marked validation_inconsistent.
5. Executor-run audit package and version audit package preserve the failed/inconsistent evidence.
6. get_audit_package validation payload exposes the new validation truth fields.
7. Redaction/bounding is preserved for stdout/stderr excerpts.
