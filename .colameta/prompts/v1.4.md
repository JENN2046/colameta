PLATFORM_BLOCKED_OPERATOR_HANDOFF_SCHEMA_VALIDATOR_V1

Goal:
Implement the first code slice for the Platform-Blocked Operator Handoff RFC by adding schema and validation logic only. This slice must not execute any operator handoff, must not add Web or MCP surfaces, and must not modify existing action dispatchers.

Required implementation:
1. Create runner/operator_handoff.py.
2. Create tests/test_operator_handoff.py.
3. Optionally update docs/platform-blocked-operator-handoff.md with a short implementation-status note only; do not rewrite the RFC.

Implementation scope:
- Define supported handoff candidate actions:
  - continue_next_version
  - apply_preview
  - run_once
  - manual_validation_apply
- Define existing guarded-only action:
  - commit_apply
- Define explicitly excluded actions and aliases:
  - push_apply
  - fetch_apply
  - pull_apply
  - restore_file_apply
  - revert_apply
  - tag
  - release
  - force_push
  - web_remote_git_mutation
  - shell_fallback
  - manual_state_edit
  - unbound_apply
- Define closeout outcome constants or enum values matching the RFC taxonomy:
  - verified_success
  - verified_noop
  - blocked_platform
  - blocked_no_operator_surface
  - blocked_receipt_missing
  - blocked_receipt_invalid
  - blocked_verification_unavailable
  - blocked_verification_mismatch
  - blocked_excluded_action
- Provide functions that validate OPERATOR_HANDOFF manifest dictionaries.
- Provide functions that validate OPERATOR_RECEIPT dictionaries against a manifest.
- Validation must treat a receipt as a claim, not proof.
- Validation must require read-only closeout verification to be declared before success can be considered valid.
- Validation must require action-specific operator surfaces.
- Validation must reject generic shell, generic file editor, generic state editor, and generic HTTP client surfaces.
- Validation must reject any remote_git_mutation=true value.
- Validation must reject secrets_included=true.
- Validation must reject excluded actions even if the receipt claims success.
- Validation must reject action mismatch, handoff_id mismatch, and operator surface mismatch.

Suggested API shape:
- validate_handoff_manifest(manifest: dict) -> dict
- validate_operator_receipt(receipt: dict, manifest: dict) -> dict
- is_excluded_action(action_class: str) -> bool
- is_supported_candidate_action(action_class: str) -> bool
- normalize_action_class(value: str) -> str
- constants for supported, guarded-only, and excluded action classes

Design constraints:
- Keep the validator pure and deterministic.
- No filesystem writes.
- No subprocess calls.
- No network calls.
- No Web Console route additions.
- No MCP tool additions.
- No executor invocation.
- No Git operation.
- No state.json mutation.
- No preview/apply execution.
- Do not import Web Console, MCP server, Git managers, executor workflow, or project patch manager.
- Standard library only unless an existing project-local utility is clearly needed.

3. test_receipt_is_claim_not_proof
4. test_excluded_push_apply_rejected
5. test_fetch_pull_rejected
6. test_commit_apply_existing_guarded_only
7. test_remote_git_mutation_true_rejected
8. test_shell_surface_rejected
9. test_generic_http_client_surface_rejected
10. test_action_mismatch_rejected
11. test_handoff_id_mismatch_rejected
12. test_operator_surface_mismatch_rejected

Manual acceptance:
- The slice only adds a pure validator and tests.
- No Web Console route is added.
- No MCP action is added.
- No operator execution surface is added.
- No remote Git mutation surface is added.
- Existing Web remote Git mutation prohibition remains untouched.
- Existing dangerous action guard remains untouched.
- Existing executor workflow remains untouched.
- Existing plan/version/patch apply code remains untouched.
