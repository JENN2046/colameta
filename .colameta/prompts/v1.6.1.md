# v1.6.1 State Lineage Reconciliation Support V1

## Goal

Add a dedicated preview-first Runner state lineage reconciliation mechanism that binds manually controlled completion receipts back into `.colameta/state.json` without running the executor and without mutating source files, Git remotes, service lifecycle, or runtime processes.

This slice exists because Git reality and Runner state can diverge after controlled/manual completion flows. The immediate target case is:

- v1.5 completed and pushed at `582f047c0989e84fd62d068dede7fa6cbbd4f7fc` with commit subject `feat(runtime): add read-only version observability`.
- v1.6 completed and pushed at `8fb86aa864a9fe98512398466fa082eb59cd60ac` with commit subject `docs(runtime): add version status decision contract`.
- v1.7 plan/prompt completed and pushed at `cd591bc0902149ad724f6d52aae29789e5fcdb54` with commit subject `chore(plan): add v1.7 runtime decision evaluator slice`.
- Runner state still reports `current_version=v1.5`, top-level status `BLOCKED_BY_ACCEPTANCE_FAILURE`, v1.5 `FAILED_BLOCKED`, v1.6 `NOT_STARTED`, v1.7 `NOT_STARTED`, and v1.5 commit metadata incorrectly points at the v1.7 plan/prompt commit.

## Required Capability

Implement two managed actions on the executor workflow reconciliation surface:

- `state_lineage_reconciliation_preview`
- `state_lineage_reconciliation_apply`

The preview action must validate a proposed reconciliation payload and produce an auditable preview artifact. The apply action must require a matching `preview_id` and must fail closed if state, HEAD, or worktree facts changed since preview.

## Required Preview Inputs

The preview input must support a bounded list of version bindings. Each binding must include:

- version
- target status, normally `PASSED` for completed versions or `NOT_STARTED` for future versions
- accepted commit hash
- accepted commit subject
- receipt or manual evidence summary
- reason for manual reconciliation

The preview must also bind:

- expected current HEAD
- expected worktree clean status
- expected branch
- expected project root or project identity
- target next runnable version

## Required Fail-Closed Checks

Preview and apply must fail closed when:

- worktree is dirty
- current HEAD does not match expected HEAD
- target version is missing from plan
- accepted commit is not in local Git history
- accepted commit subject does not match the supplied subject
- required evidence fields are missing
- the target transition skips an unresolved earlier version without an explicit binding for that earlier version
- the payload attempts to mark a future version as completed without evidence
- the payload attempts to modify source files, run executor, push, fetch, pull, tag, release, restart, reload, or kill processes

## Required State Behavior

For the immediate target reconciliation, the preview should be able to propose:

- mark v1.5 as `PASSED` and bind it to commit `582f047c0989e84fd62d068dede7fa6cbbd4f7fc`
- mark v1.6 as `PASSED` and bind it to commit `8fb86aa864a9fe98512398466fa082eb59cd60ac`
- keep v1.7 as `NOT_STARTED`
- set the top-level Runner status and current/next version fields so that v1.7 becomes the next runnable version
- remove or supersede the incorrect v1.5 commit metadata pointing to `cd591bc0902149ad724f6d52aae29789e5fcdb54`

Apply must only touch `.colameta/state.json` and controlled preview bookkeeping. It must not modify source files, docs, tests, plan, prompt, runtime implementation files, Git commits, tags, remotes, service lifecycle state, executor sessions, or process lifecycle.

## Audit Requirements

Preview and apply responses must include an auditable before/after summary:

- old top-level Runner status
- new top-level Runner status
- old current version/status
- new current version/status
- old next version/status
- new next version/status
- per-version old/new status for each binding
- per-version old/new commit metadata for each binding
- expected current HEAD
- evidence summaries, redacted for sensitive values
- files that would be touched by apply
- warnings and blockers

The preview artifact must be one-use and should expire like existing executor workflow previews.

## Tests

Add focused unit tests for:

- preview generation for v1.5/v1.6 completed and v1.7 not started
- apply requiring preview id
- apply only changing expected state fields
- dirty worktree fail-closed behavior
- HEAD mismatch fail-closed behavior
- missing plan version fail-closed behavior
- missing local commit fail-closed behavior
- missing evidence fail-closed behavior
- unresolved earlier version skip fail-closed behavior
- no executor run, Git remote operation, restart, reload, kill, or source file mutation is performed

## Out of Scope

- Executor run or bounded loop
- Source implementation for runtime version decision evaluation
- Web Console route changes
- Git remote mutation
- service lifecycle mutation
- restart/reload/kill/apply process lifecycle behavior
- direct manual editing of `.colameta/state.json`
- commit, push, tag, or release
