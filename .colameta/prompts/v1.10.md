This can be misleading. The mismatch may represent stale historical executor session or resume metadata, not an executor actively running on an old HEAD.

## Goal

Implement a small read-only classification rule for executor-session head mismatch.

The system should clearly distinguish between:

1. An active operation whose recorded session HEAD differs from the current project HEAD.
2. A completed idle stale session where old resume metadata remains after the project has already completed and advanced to a new HEAD.
3. An unknown or incomplete mismatch where evidence is insufficient and the system should fail closed.

This task must not clean the session, restart services, reload code, kill processes, start an executor, commit, push, or mutate remote state.

## Classification Rules

### Rule A: No Mismatch

If there is no executor session, or the recorded session HEAD matches the current Git HEAD:

- status = none
- severity = info
- no blocker should be added
- no high-risk warning should be emitted

### Rule B: Active Operation Head Mismatch

If operation_running is true, or the live claim/run is still running, and the recorded session HEAD differs from the current Git HEAD:

- status = active_operation_head_mismatch
- severity = blocked
- blocks_auto_resume = true
- blocks_auto_start = true
- require human judgment
- do not modify runtime/session files automatically

### Rule C: Completed Idle Stale Session

If all of the following are true:

- executor session exists
- recorded session HEAD differs from current Git HEAD
- operation_running is false
- job status is idle
- latest run or claim is completed
- Runner state is VERSION_PASSED and current version is PASSED
- worktree is clean

Then:

- status = completed_idle_stale_session
- severity = warning
- blocks_auto_resume = true, or at minimum should_resume = false
- blocks_auto_start = false
- do not display this as an active operation
- do not require automatic cleanup
- do not authorize restart, reload, kill, apply, commit, or push

### Rule D: Unknown or Incomplete Evidence

If the recorded session HEAD differs from the current Git HEAD, but the system cannot confirm operation/job/latest-run/state/worktree evidence:

- status = unknown_head_mismatch
- severity = warning or blocked, following existing fail-closed safety conventions
- blocks_auto_resume = true
- auto-start behavior should follow existing Runner policy
- operator_message must explain that evidence is incomplete

## Display Requirements

Expose the classification on at least one existing status surface, preferably one or more of:

- /api/status
- /api/v2/status
- executor continuation preview/decision
- MCP manage_executor_workflow status

Do not add any new dangerous mutation route.

Status output and UI text must avoid showing only the raw label "head_mismatch". The operator should be able to tell whether the mismatch is historical stale session metadata or an active operation mismatch.

A completed_idle_stale_session must not be displayed as an operation currently running.

## Tests

Add or extend focused tests covering:

1. Session HEAD equals current HEAD, producing no mismatch classification.
2. operation_running=true with head mismatch, producing active_operation_head_mismatch and blocking automatic resume/start.
3. operation_running=false, job idle, latest run completed, state PASSED, clean worktree, and head mismatch, producing completed_idle_stale_session.
4. completed_idle_stale_session does not authorize restart, reload, kill, apply, commit, or push.
5. Incomplete evidence produces unknown_head_mismatch or an equivalent fail-closed state.
6. Status/API output includes reason or operator_message, not only a raw head_mismatch warning.
7. Existing v1.9 runtime loaded-code verification and validation truth-source behavior do not regress.
