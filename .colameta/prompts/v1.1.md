COMMANDER_TASK: E2D_GIT_COMMIT_CONFIRM_GUARD

Mode:
controlled implementation via Runner.

Project:
colameta-self-dev

Goal:
Protect the Web Console local Git commit confirmation route with the same dangerous-action preview-confirm-audit model used by E2C, without changing MCP Git manager internals or adding remote push/pull Web actions.

Scope:
- Add dangerous-action confirmation for /api/commit-confirm.
- /api/commit-preview remains a lower-risk runtime metadata preview and should not require dangerous confirmation in this slice.
- The actual commit execution must still use MCPGitCommitManager.commit(preview_id) and retain its internal preview/HEAD/diff/file-set guards.
- Web dangerous confirmation should run before _api_commit_confirm_with_project() dispatches to manager.commit().
- The dangerous receipt must be attached to commit-confirm responses and must redact confirmation_id.

Required behavior:
1. /api/commit-confirm without dangerous confirmation returns 403 before manager.commit() is called.
2. /api/commit-confirm with expired, reused, wrong route, wrong action, wrong payload, stale HEAD, stale runner state, stale plan, or stale commit preview signature is rejected before commit.
3. A valid dangerous confirmation allows the existing commit confirm flow to call manager.commit() exactly once.
4. The dangerous_action_receipt is included on commit-confirm responses and redacts confirmation_id.
5. Full confirmation id must not leak in response payloads.
6. Existing E2C Web plan/state dangerous-action tests continue to pass.
7. Existing commit preview/confirm behavior remains compatible except for the new confirmation requirement on commit-confirm.
8. Web remote Git status remains read-only; do not add Web push/pull apply routes.

Suggested implementation notes:
- Add /api/commit-confirm to DANGEROUS_WEB_CONFIRMATION_ROUTES.
- Add a Git local history policy, for example action_type=git_commit_confirm and risk_class=git_local_history_action.
- Bind confirmation context to current HEAD, runner state signature, plan signature, and a commit preview signature.
- The commit preview signature should represent the pending Web commit preview that _api_commit_confirm will use, including preview_id, diff_hash, commit_files, version, and project_root where available.
- Do not print or return the actual confirmation id after apply; only REDACTED may appear in receipts.
- Keep /api/commit-preview outside dangerous confirmation for this slice, but document that commit-confirm is the Git history write boundary.
