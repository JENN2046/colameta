## 要修复的问题

v1.7 materialized audit package 报告 `validation_status_summary=passed`，并包含
`PASSED: python3 -m unittest discover -s tests`。但 executor report excerpt 明确说明
同一命令因为 38 个 socket `PermissionError` 失败。

这意味着 report/audit normalization 允许失败或不一致的 validation command 被总结成
passed。

## 目标

加固 executor validation reporting，确保 failed command evidence 不能被归一化成 PASS。

`Truth-Source` = 真相源。中文意思是：验收到底过没过，以结构化命令记录和 exit code
为主，而不是以漂亮的摘要字符串为主。

## 真相源规则

1. Structured command records 和 exit codes 是 primary source of truth。
2. 如果 command record 的 `exit_code` 非零，validation 不能总结为 passed，即使
   status string 写着 PASSED。
3. 如果 status 和 exit_code 冲突，必须保留明确的 inconsistency signal。
4. 如果 final executor report text 和 structured validation summary 对同一命令互相
   矛盾，必须保留 `validation_inconsistent`，不能静默接受 summary。
5. Audit packages 和 `get_audit_package` validation reads 必须清楚暴露 failed 或
   inconsistent command evidence，足以供 Commander 审查。

## 必需实现行为

- 增加或重构 pure helper logic；如果能保持 `runner/executor_run_reports.py` 可读，优先
  放在小而聚焦的模块中。
- 保留旧的 `summary.validation_results` string lists 兼容性。
- 在 reports/audit packages 中加入 structured validation command records，至少包含：
  - command index 或 stable id；
  - original command text；
  - executed command text；
  - recorded status；
  - derived status from exit_code；
  - exit_code；
  - bounded/redacted stdout/stderr excerpts；
  - status 和 exit_code 冲突时的 inconsistency reason。
- 有结构化 evidence 时，从它推导 `validation_status_summary`。
- 任意 command 的 `exit_code != 0` 时，summary 必须是 failed 或 inconsistent，绝不能是 passed。
- status 写 PASSED 但 exit_code 非零时，`validation_inconsistent` 必须为 true。
- summary text 写 PASSED 但 executor report text 显示同一命令失败时，`validation_inconsistent`
  必须为 true。
- materialized executor-run audit package 和 version audit package 必须携带新 truth-source fields。
- `get_audit_package(section="validation")` 应暴露 structured command records 或 bounded sample
  加 `validation_inconsistent`，不能只有 plain string samples。
- Markdown report output 在可用时应包含 validation command 的 exit_code。

## 测试优先

新增聚焦测试，至少覆盖：

1. status=PASSED 且 exit_code=1 的 synthetic command record 会产生
   `validation_inconsistent=true`，并且 summary 不等于 passed。
2. status=FAILED 且 exit_code=1 会总结为 failed，并保留 command、exit_code、stdout/stderr excerpts。
3. 旧 string-only `validation_results` fallback 仍可工作。
4. summary 写 `PASSED: python3 -m unittest discover -s tests` 但 report text 显示同命令失败时，
   标记为 `validation_inconsistent`。
5. executor-run audit package 和 version audit package 保留 failed/inconsistent evidence。
6. `get_audit_package` validation payload 暴露新的 validation truth fields。
7. stdout/stderr excerpts 保持 redaction/bounding。

## Closeout 要求

最终报告必须列出 changed files、每条 validation command 及 exit code。如果完整 unittest
discovery 因环境 socket PermissionError 失败，不能称为 passed；只有 targeted tests 通过且
socket-permitted local shell 另行恢复时，才能把它分类为平台/环境 caveat。

最终摘要必须明确说明 `validation_inconsistent` protection 是否已测试。除非单独请求
controlled commit preview，否则停在 commit/push 前。
