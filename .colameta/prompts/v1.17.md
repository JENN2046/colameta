Implement v1.17: Connector Tunnel Evidence Receipt And Closeout Packet V1.

Goal:
Add a bounded receipt and closeout packet for real connector/tunnel usability checks. This version should make the external connector evidence trail reviewable without reading secrets or mutating connector, tunnel, proxy, provider, service, Git, executor, or delivery state.

Required behavior:
- Produce a concise connector/tunnel closeout receipt format that records commit/runtime refs, local Web/MCP health, runtime freshness, tunnel-client status, tunnel control-plane status, evidence gaps, and closeout decision.
- Use only sanitized evidence from approved status surfaces. Do not read tunnel-client config, tunnel-client logs, runtime keys, proxy/provider config, tokens, cookies, credentials, private memory, or provider raw responses.
- Preserve the distinction between `healthy`, `degraded`, `unavailable`, and `unverified` for both tunnel-client and control-plane evidence.
- Keep closeout blocked when either tunnel-client or control-plane evidence is missing or degraded.
- Make the packet useful for Web GPT, Codex, planner, reviewer, and Jenn to decide whether connector/tunnel verification is complete or still blocked.

Scope:
Allowed files are exactly the plan allowed_files. Receipt files must be short, factual, and free of raw secrets or provider responses.

Forbidden actions:
Do not automatically repair connector/tunnel/proxy/provider state, restart tunnel-client, replace stable service, run executor, route transition, create ReviewDecision, emit GateEvent, write Delivery State accepted, push, deploy, release, or publish.

Validation:
Run the acceptance commands from the plan. Review the receipt text for secret-bearing strings before reporting completion.
