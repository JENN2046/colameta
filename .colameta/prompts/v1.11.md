Implement v1.11: Connector / runtime health observability.

Goal:
Add a read-only observability layer that lets ColaMeta users distinguish these cases quickly:
1. ColaMeta stable service is down or PID/ports do not match status.
2. Local MCP/Web service is healthy, but the external MCP connector/tunnel path is degraded.
3. tunnel-client is locally live/ready but its OpenAI control-plane poll path is failing.
4. Runtime loaded code/provenance is healthy, but connector availability depends on external network/proxy state.

Required behavior:
- Keep all diagnostics read-only.
- Reuse existing local patterns in runtime_observability, mcp_server, web_console, service_lifecycle_store, and runner_cli.
- Add structured fields that summarize connector/runtime health without exposing secrets:
  - local service PID/port health source and whether it was discovered from metadata or process table.
  - MCP/Web health status.
  - tunnel/client health availability if safely discoverable.
  - control-plane reachability status if safely inferable from non-secret health/log/status surfaces.
  - clear reason codes such as LOCAL_SERVICE_HEALTHY, MCP_ENDPOINT_HEALTHY, TUNNEL_CLIENT_UNAVAILABLE, TUNNEL_CONTROL_PLANE_DEGRADED, CONNECTOR_HEALTH_UNVERIFIED, or equivalent local naming.
- Fail closed when evidence is unavailable. Do not infer success from missing tunnel-client data.
- Ensure outputs explicitly separate local ColaMeta service health from external connector health.
- Avoid reading or printing secrets, tokens, cookies, subscription URLs, provider raw responses, or private memory.
- Do not modify auth/provider/tunnel/proxy/global Codex configuration.

Expected user-facing surfaces:
- Existing runtime/version/status output should make the local-vs-external connector boundary visible.
- Existing MCP read-only status/readiness tools should include compact connector/runtime health evidence where appropriate.
- Web/API status may include the same safe summary if it already has a matching status surface.

Scope:
Allowed files are exactly the plan allowed_files. Do not touch forbidden files. Keep the implementation narrow and compatible with current stable service behavior.

Validation:
Run the acceptance commands exactly. Report changed files, validation results, and residual risks. Do not commit.
