Implement v1.16: Connector Runtime Health MCP Closeout Tool V1.

Goal:
Expose a read-only MCP connector/runtime health closeout tool that can distinguish local ColaMeta runtime health from real external connector/tunnel usability evidence. The tool must fail closed when tunnel-client or tunnel control-plane evidence is missing.

Required behavior:
- Add `get_connector_runtime_health_status` as a normal MCP read tool with `mcp:read`, `read_only=true`, and `side_effects=false`.
- Combine runtime freshness from `get_runtime_version_status`, local Web/MCP service evidence, and optional caller-provided sanitized `tunnel_client` / `control_plane` evidence.
- Accept only these external evidence fields: `status`, `reason_code`, `evidence_source`, `last_observed_at`.
- Reject or sanitize anything outside the bounded evidence shape; never echo raw token, cookie, credential, provider auth, provider raw response, proxy config, or tunnel-client config content.
- Keep closeout blocked as `local_runtime_ready_external_connector_unverified` until runtime, local service, tunnel-client, and control-plane evidence are all healthy.
- Return `connector_closeout_ready` only when local runtime, Web/MCP, tunnel-client, and control-plane evidence are healthy.
- Add the tool to Web GPT / local Codex service entry guidance so external connector closeout is visible during normal first reads.
- Keep `colameta status` aligned with runtime freshness so the CLI closeout line does not report stale local/runtime facts.

Scope:
Allowed files are exactly the plan allowed_files. Do not read or write tunnel-client config/logs/runtime key, proxy/provider config, secrets, credentials, tokens, cookies, private memory, stable service files, Runner state, decisions, memory, or todolist.

Forbidden actions:
Do not repair proxy/network/provider configuration, restart tunnel-client, restart or replace stable service, call an external paid provider API, run executor, create ReviewDecision, emit GateEvent, accept Delivery State, push, deploy, release, or publish.

Validation:
Run the acceptance commands from the plan. Also verify that unsafe external evidence fields are not echoed in success or error results. Report changed files, validation results, and residual risks.
